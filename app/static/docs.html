Overview

This system receives Dahua alarm events (NotifyEvent), optionally correlates them with snapshots, and produces events. Those events can trigger automations which run actions such as logging or calling webhooks (Home Assistant / Slack / Discord / custom services).

Key URLs

  • / – Main site
  • /settings.html – Edit settings.json
  • /automations.html – Edit automations.json
  • /api/settings – Read/write settings
  • /api/automations – Read/write automations
  • /api/health – Health check

Events

Dahua sends events as HTTP-ish POSTs to the alarm listener on port 15000. Each detection usually generates two messages:

  • "Action": "Start" when the event begins
  • "Action": "Stop" when the event ends

Event fields

Events are JSON objects with fields like:

{
  "id": "2026-01-09T12:45:15_172.16.0.100_ch5",
  "timestamp": "2026-01-09T12:45:15",
  "camera_ip": "172.16.0.100",
  "action": "Start",
  "code": "FaceDetection",
  "locale_time": "2026-01-09 12:45:15"
}
Tip: If you only want a single event per detection, configure: only_start_events: true to ignore Stop events.

Automation System

Automations are stored in data/automations.json. Each automation contains:

  • Trigger – which events it reacts to
  • Conditions – optional logic filters
  • Actions – what to do (log, webhook, etc.)

Automation schema

{
  "id": "notify-on-face",
  "name": "Notify on Face Detection",
  "enabled": true,
  "trigger": {
    "event_code": "FaceDetection",
    "action": "Start"
  },
  "conditions": [
    { "field": "camera_ip", "op": "equals", "value": "172.16.0.100" }
  ],
  "actions": [
    { "type": "log", "message": "Face detected at {{locale_time}}" }
  ]
}

Triggers

  • event_code → matches event field code
  • action → matches event field action (Start/Stop)

Conditions

Each condition is evaluated against the event object.

  • equals / not_equals
  • contains (strings only)
  • in (value must be list)
  • exists (field present + not null)

Actions

  • log – prints to docker logs
  • webhook – HTTP request with JSON payload
Note: If you change listener host/port, you may need to restart the alarm listener process. The Settings page can include a restart button.

Examples

1) Log every FaceDetection Start

{
  "id": "log-face-start",
  "name": "Log FaceDetection Start",
  "enabled": true,
  "trigger": { "event_code": "FaceDetection", "action": "Start" },
  "conditions": [],
  "actions": [
    { "type": "log", "message": "🙂 Face detected on {{camera_ip}} at {{locale_time}}" }
  ]
}

2) Send webhook to Home Assistant

{
  "id": "ha-face-webhook",
  "name": "Send FaceDetection to Home Assistant",
  "enabled": true,
  "trigger": { "event_code": "FaceDetection", "action": "Start" },
  "conditions": [],
  "actions": [
    {
      "type": "webhook",
      "url": "http://homeassistant.local:8123/api/webhook/abcd1234",
      "method": "POST",
      "payload": {
        "type": "{{code}}",
        "camera_ip": "{{camera_ip}}",
        "time": "{{locale_time}}"
      }
    }
  ]
}

Storage & Logs

All persistent data lives under /app/data inside the container. If you run docker with: 

-v "$HOME/sei-raspi/data:/app/data"

…then your data files are stored on the host at:

  • ~/sei-raspi/data/settings.json
  • ~/sei-raspi/data/automations.json
  • ~/sei-raspi/data/events.jsonl (if enabled)
  • ~/sei-raspi/data/automation_runs.jsonl

View logs

docker logs -f sei-raspi

Tail automation runs

tail -f ~/sei-raspi/data/automation_runs.jsonl

Troubleshooting

  • Events truncated: ensure alarm listener reads full payload based on Content-Length.
  • Two events per detection: Dahua sends Start and Stop; enable only_start_events to avoid duplicates.
  • settings.json missing: load_settings() should create defaults on first run, or create the file manually.
  • Port 15000 not reachable: ensure container starts with -p 15000:15000.
